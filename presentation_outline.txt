Presentation Outline: Model / Service / Repository
====================================================

1. MODEL LAYER — Separation of Concerns
----------------------------------------
- Three distinct model classes, each with a clear responsibility:
    - CodeBlock: a single line/block of code
        - Fields: isDistractor | orderIndex | Code Content
        - Distractors (wrong answers) vs. solution blocks distinguished here
    - ParsonsProblem: the aggregate root
        - Owns a List<CodeBlock>; contains title and instructions
        - null-check enforced in setCode() to prevent invalid state
    - ProblemsList: XML root wrapper
        - Exists purely to satisfy Jackson's need for a root element
        - No business logic; purely structural

- Models are plain POJOs — no inheritance, no business logic
- Talk about why: keeps the model focused on data shape, everything else lives elsewhere

3. REPOSITORY LAYER — Jackson XML Mapping
------------------------------------------
- IParsonsProblemsRepository defines the contract (interface):
    - save, findById, findAll, existsById, deleteById
    - Interface-based so the implementation is swappable

- XmlParsonsProblemsRepository is the concrete implementation:
    - Reads/writes to a local XML file (DemoRepo.xml) using Jackson's XmlMapper
    - Two private helpers keep I/O isolated:
        - readFromRepo(): deserializes XML → ProblemsList, handles missing file gracefully
        - writeToRepo(): serializes ProblemsList → XML

- Jackson annotations on the model drive the XML structure:
    - @JacksonXmlRootElement — sets the XML tag name for each class
    - @JacksonXmlElementWrapper — wraps the list of blocks in a <blocks> tag
    - @JacksonXmlProperty — names each <block> element

- ID assignment lives in the repo:
    - New problems (id == -1) get assigned max(existing IDs) + 1
    - Existing problems are updated in place

- Example XML output:
    <Problems>
      <Problem>
        <id>1</id>
        <title>Check if a Number is Positive</title>
        <blocks>
          <block>
            <codeContent>def is_positive(n):</codeContent>
            <isDistractor>false</isDistractor>
            <orderIndex>0</orderIndex>
          </block>
        </blocks>
      </Problem>
    </Problems>

2. SERVICE LAYER — Decoupling via Dependency Injection
-------------------------------------------------------
- ParsonsProblemsService is the business logic hub - calls all functionality for interacting with repository
- Receives an IParsonsProblemsRepository via constructor (not a concrete class)
    - This means you can swap XML for a DB or in-memory store without touching the service
    - Shown in ParsonsApplication.java where wiring happens at startup

- Key responsibilities of the service:
    - CRUD delegation to the repository (saveProblem, deleteProblem, getAllProblems, etc.)
    - Business logic that doesn't belong in the model or repo:
        - getSortedSolution(): filters out distractors, sorts remaining blocks by orderIndex
          using Java Streams (filter + sorted + toList)
        - checkAnswer(): fetches the problem, gets the sorted solution, compares the
          student's submitted answer block-by-block (size, order, content, no distractors)
    - Input validation: throws IllegalArgumentException with context if problem not found

- The service is the only layer the UI talks to — GUI/CLI never touches the repo directly
